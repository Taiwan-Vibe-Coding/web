課程日期：2026-07-31（五）・結業堂

練習一：複合任務查詢（30 分鐘）

課堂說明

前三堂多半是「問一句、查一件」。今天要練的是：同一個對話裡查多件事，再整合成一份給主管看的摘要。

Session 3 課後作業寫過「整合式提問」——今天把它變成一份真正能用的週一快報。

情境

每週一早上，主管要一份簡短的「公司現況快報」。以前要開好幾個畫面慢慢抄，今天請 Codex 一次查完、整理好。

Step 1：用自己的話先問（8 分鐘）

不要照抄下面的範本。用你平常會跟同事講的話，請 Codex 做一份本週快報。

我的問法：

Codex 大概查了哪些方向？（休假／採購／帳款／其他）

這份結果可以直接轉給主管嗎？
□ 可以
□ 要改格式
□ 內容有問題：

Step 2：換成結構化問法（12 分鐘）

把同一件事改成下面這種「分區＋要什麼欄位」的問法，再跑一次，跟 Step 1 比較。

請幫我整理一份穩得本週業務快報，格式如下：

【人力】
- 本週有哪些同仁在休假？各休幾天？

【採購】
- 有哪些採購單需要關注？（預計到貨已過、或金額比較大的）

【應付】
- 近期有哪些應付帳款還沒付？哪一筆最急？總金額大約多少？

【應收】
- 有哪些應收帳款還沒收到？有沒有逾期的？

【建議行動】
- 根據上面資料，列出 1–3 件這週該處理的事（需人工決策的標成「待主管決定」）

繁體中文，條列，每項不超過兩行。
兩次差在哪裡？

哪個部分差最多？是問法，還是系統資料本身？

Step 3：同一個對話再追問一次（7 分鐘）

選一個追問（不要三個都做）：

A. 排出優先順序

剛才那份快報裡，最需要今天處理的是哪一件？為什麼？

B. 換成英文

把這份快報改成英文版，給外籍顧問看。

C. 想自動化

如果這份快報每週一早上要自動出現，我大概要做什麼設定？
我選的是：A／B／C

Codex 的回答有用嗎？還缺什麼？

討論問題（3 分鐘）

  1. 「建議行動」是 Codex 該下結論，還是人該下結論？
  2. 如果快報裡有一筆資料明顯過時，你會怎麼處理——改問法，還是先通報資料負責人？

練習二：寫你的 System Prompt（40 分鐘）

課堂說明

這是四堂課的技巧 4／4：System Prompt。把 RCFC 裡常重複的 Role／Context 固定下來，之後提問專注 Format 與 Constraint。寫好後用同一句「有哪些採購單需要處理？」對比有／無 System Prompt 的差異。

什麼是 System Prompt？

System Prompt 是你在每次對話開始前，告訴 Codex 的背景設定。 寫好之後，Codex 每次回答都會帶著這個背景，不用每次都解釋。

沒有 System Prompt：

你問：「有沒有快到期的款項？」
Codex 可能問：「你指的是哪種款項？應付還是應收？哪家公司？」

有 System Prompt：

Codex 直接查應付帳款，用台幣顯示，標出最快到期的，用中文條列回傳。

Step 3：在 Codex 上把「凡事問AI」跑起來（15 分鐘）

把 Step 2「系統可以協助」的前 1～3 件事，真的在 Codex 問一次。

1. 接受回答前先確認

不要直接把數字貼進報告。至少想過：

  • 期間／對象對不對（本週？哪個客戶／廠商／員工？）
  • 金額或筆數有沒有明顯不合理
  • 需不需要再對正航／鼎新或原始單據

2. 資料看起來不對時

先找同組或主管確認；確定是 ERP 資料本身的問題，再通報 MIS。

3.（選做）排進每週一

左側 Scheduled → 選 Weekly review（或類似範本）→ 把你「每週一早上」要問的 1～3 句貼進去，設成每週一。

練習四：讓 Codex 安全地「做事」——建立你自己的客戶與報價單（25 分鐘）

課堂說明

這四堂課多半在「查資料」。這個練習在業務資料上再走一次「預覽 → 確認」：建立屬於自己的客戶，再幫這個客戶建報價單。客戶名稱加自己的姓名英文縮寫前綴（例如 YL_測試客戶）。

為什麼要做這個練習

前三堂課，Codex 做的都是「查資料」。Session 2 你申請過一次假單，體驗過「預覽 → 確認」——那是目前唯一一次讓 Codex「做事」而不只是「查資料」。

其實 O2C／P2P 有一整套這樣的寫入工具（建立客戶、建立訂單、建立報價單、確認報價⋯），今天用其中兩個，讓大家再次體驗這個安全機制——這次是在「業務資料」上，而不是「我自己的假單」，而且是建立一個完全屬於你自己的新客戶，不是去動大家共用的山田電機。

今天的規則：姓名前綴，不是「只能預覽」。全班都真的走完「預覽 → 確認」兩步。差別在於：你建立的客戶名稱前面一律加上你自己的姓名英文縮寫（例如 YL_測試客戶）。這樣不管全班有幾個人同時操作，彼此的資料都不會撞名、不會互相蓋掉，list_customers 之後會看到一排 YL_、WCH_ 這樣的前綴，一眼就能分辨是誰的練習資料。
