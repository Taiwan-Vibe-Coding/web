Day 3 · Session 3 · 課程計畫

採購與財務的 AI 查詢助理

公司資料整合查詢

日期：2026-07-28（二）

時長：3 小時

對象：行政採購、財務、管理層、實驗室、業務

工具：ChatGPT Work（Codex）+ 穩得 MCP 外掛（示範 ERP）

需要 ERP 連線：✅ 需要

講師：陳育亮教授

稱呼：ChatGPT Work＝App；Codex＝派任務能力；穩得 MCP 外掛＝連示範 ERP。

課程目標

學員結束時能做到：

  1. 用自然語言查詢廠商資料、採購單狀態、應付帳款明細
  2. 讓 Codex 把多個查詢結果串起來分析，不用自己在 Excel 彙整
  3. 說出「Codex 整理資料」和「Codex 做決定」的差別
  4. 識別哪些判斷必須由人負責，Codex 只能提供資訊

各節詳細內容

13:00–13:15 作業分享 + 角色轉換（15 分鐘）

作業分享（8 分鐘）：Session 2 的「我想查的公司資料」

請 2-3 位學員分享作業：自己部門每週要查的公司資料、用 RCFC 寫的問法草稿。

講師記下大家提到的資料需求——待會練習如果剛好對得上，可以直接拿出來驗證；對不上的，記下來留給 Session 4 的「凡事問AI」練習用。

回顧（7 分鐘）：

「上次大家用 Codex 查了自己的假單和薪資。對『預覽 → 確認』這個機制的想法有沒有改變？」

今天的轉換：

Session 2 我們查的是「我的」資料——我的假、我的薪資。 那些是個人隱私，系統限制你只能查自己的。

今天我們查的是「公司的」資料——廠商、採購單、應付帳款。 這些是公開給有授權的同事看的業務資料。

查詢的工具一樣，但資料的性質不同，使用的責任也不同。

今天全程唯讀，沒有任何寫入練習。 PO2026-001、PO2026-002、C001（山田電機）、C003（台達）、V003（全誠校正）、V004（SGS 台灣）、V005（承澤科技）這些是今天跟下堂課反覆對照用的共用示範資料。

13:15–13:40 概念：AI 整理資料，人做決定（25 分鐘）

這堂課要解決的問題

行政採購同事平常的一個典型工作流：

問題：「這張採購單付款了嗎？付款條件是什麼？廠商的聯絡人是誰？」

原本的做法：

  1. 打開 ERP 找採購單
  2. 切換到廠商主檔找聯絡人
  3. 切換到應付帳款確認付款狀態
  4. 三個畫面來回，手動彙整

用 Codex 的做法：

PO2026-001 付款了嗎？廠商的聯絡人和付款條件是什麼？

→ Codex 查三個地方，整理成一段話回傳

現場 Demo（講師操作）：

示範一：

列出所有廠商，以付款條件分組。

示範二：

PO2026-002 的狀態是什麼？預計到貨日期是？廠商是誰？聯絡人是？

示範三：

目前有哪些應付帳款還沒付？總金額是多少？哪一筆最快到期？

觀察重點：

  • Codex 是否在一個回答裡整合了多個查詢的結果？
  • 回傳的格式是否比 ERP 畫面更容易閱讀？
  • Codex 有沒有「建議要趕快付哪筆」？（如果有，這是好事還是壞事？）

「整理」和「決定」的界線

Codex 能做的 Codex 不應該做的
列出所有逾期未付的發票 決定要先付哪一筆
計算總應付金額 判斷是否要要求廠商延期
找出付款期限最近的廠商 代表公司與廠商溝通
把資料整理成報告格式 批准任何付款動作

Codex 是你的資料助理，不是你的財務主管。 決定怎麼做，永遠是人的責任。

這是四堂課的技巧 3／4：整合式提問。 上面「示範二」「示範三」都是一次問裡藏了好幾件事（廠商＋狀態＋交期＋聯絡人）。技巧不是問得更複雜，是在一句話裡把你原本要跑三個畫面才能兜起來的資訊講清楚——Codex 會自己去查、自己整合，你只要審核結果對不對。

13:40–15:40 動手練習

詳見 session-03-exercises

  • 練習一（30 分鐘）：查廠商
  • 練習二（30 分鐘）：查採購單
  • 練習三（25 分鐘）：查應付帳款
  • 練習四（25 分鐘）：查客戶與委託案（O2C）
  • 延伸練習五～八（選修，視時間；優先序見 exercises）：數據分析／發票明細／庫存／專案與合約／電子發票概念

練習規則：

  1. 先用自己的話問，不要照格式
  2. 記下 Codex 查了哪些資料來源
  3. 試著讓 Codex「串接」多個查詢
  4. 討論問題請認真作答，沒有標準答案
  5. 今天全程唯讀

15:40–15:50 討論（10 分鐘）

「今天 Codex 幫你查了廠商、採購單、應付帳款、客戶委託案。有沒有哪個查詢結果，讓你有點擔心？擔心什麼？」

幾個可能方向：

  • 「萬一 Codex 的計算有錯？」→ 確認方式：直接核對原始發票
  • 「有授權問題嗎？誰可以查這些資料？」→ 這是 MIS 要設定的存取控制
  • 「如果資料不對，要找誰？」→ 找 ERP 管理員更新來源資料

15:50–16:00 預告 Session 4 + 作業指派（10 分鐘）

前三堂課，我們一直在「查」。

最後一堂，我們要問一個更大的問題： Codex 能不能變成穩得自己的 AI 同事？

什麼叫自己的 AI 同事？就是它知道穩得是做什麼的、它知道你們的業務流程、它記得你常問什麼。

Session 4 會教大家怎麼寫「讓 Codex 記住你是誰」的 System Prompt， 以及怎麼設計一個讓自己部門每週都能固定用的 Codex 習慣。

課後作業（必做，Session 4 開場分享）

用今天學到的「整合式提問」，把 Session 2 作業裡你寫的問法，改寫成「一次問多件事」的版本（例如：把三個分開的問題合併成一句）。帶著這個問法草稿來 Session 4，「凡事問AI」練習會直接用到。

本堂講義重點（帶回去用）

Codex 能整理 vs. 不應該做的決定

計算總應付金額 判斷是否要求廠商延期

其他的示範問法清單（照樣造句用）：

有哪些客戶？列出名稱與國別。
山田電機目前有哪些委託案？進度到哪了？
台達目前有哪些委託案？應收帳款還有沒有沒收的？
目前有哪些應收帳款還沒收到款？有沒有逾期的？
SINV-2026-041 這張發票的內容是什麼？金額多少？狀態是？
PO2026-001 的狀態是什麼？預計到貨日期是？已經過期了嗎？
幫我列出所有採購單，以及每張單對應的廠商名稱、付款條件和聯絡人，整理成表格。
SGS 台灣有沒有未付的發票？發票號是什麼？
哪些品項的庫存低於再訂購點？
幫我做一個付款追蹤摘要：未付應付帳款、對應採購單、哪筆最緊急。

提醒：Codex 只能回傳 ERP 裡記錄的資料。ERP 的資料品質，決定了 AI 助理能給你的答案品質——資料沒更新，Codex 也看不到真實狀況。
