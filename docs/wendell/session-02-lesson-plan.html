我的 AI 人資助理：Codex 直接查你的 ERP 資料

AI 人資助理＋MCP

日期：2026-07-24（五）

時長：3 小時

對象：全體學員（MIS + 實驗室 + 業務 + 行政）

工具：ChatGPT Work（Codex）+ 穩得 MCP 外掛（示範 ERP）

需要 ERP 連線：✅ 需要（GCP 示範 ERP／Dataverse，非正航/鼎新正式資料）

講師：陳育亮教授

稱呼：ChatGPT Work＝App；Codex＝派任務能力；穩得 MCP 外掛＝連示範 ERP。

課程目標

學員結束時能做到：

  1. 說出「Codex + MCP」和「單純問 Codex 問題」的根本差異
  2. 用自然語言查詢自己的請假紀錄、薪資明細
  3. 透過 Codex 發出請假申請，並理解「預覽 → 確認」安全機制的原因
  4. 說出哪些情境適合讓 AI 執行、哪些必須人工確認

各節詳細內容

13:00–13:15 作業分享 + 破冰（15 分鐘）

作業分享（8 分鐘）：Session 1 的 RCFC 任務描述

請 2-3 位學員分享作業：原本一句話 vs. 加上 RCFC 之後的任務描述，兩次輸出差在哪裡。

講師引導觀察：

  • 大家改寫時，最常補的是哪個要素（Role／Context／Format／Constraint）？
  • 有沒有人發現「原本以為不重要的細節」其實影響很大？

破冰（7 分鐘）：

讓學員分享：

  • 除了作業，這週還有沒有另外用 Codex 做過別的事？
  • 哪裡好用？哪裡讓你不放心？

今天的轉折：

Session 1 的 Codex 是幫你做文件工作，它靠的是語言能力。

今天的 Codex 是幫你查系統資料，它靠的是連進外部系統。

這是完全不同的一件事。

提醒：今天連的是示範環境（wendell-dataverse-cloud），資料是為了上課準備的假資料，不是穩得的正航/鼎新正式資料。如果以後公司希望將正式系統透過AI來操作，操作方式會跟以後接上時完全一樣，只是現在查到的內容不是真的公司紀錄（請大家不要把練習結果當成真實個資或帳務數字）。

13:15–13:50 概念：Codex + 工具 = 什麼？（35 分鐘）

兩種 Codex 的差別

單純 Codex Codex + MCP 工具
資料來源 訓練資料（截止到某個日期） 即時連線外部系統（今天是示範環境）
問「我今年請了幾天假？」 會亂猜或說「我不知道」 查系統資料，回傳查詢結果（今天是示範資料）
問「上個月薪水是多少？」 無法回答 查薪資系統，回傳明細（今天是示範資料；有 4／5／6 月）
適合做什麼 語言、翻譯、寫作、分析文件 查詢 ERP 資料、提交申請

先說清楚：今天查的是「我的」資料，不是「公司的」資料。 底層系統其實分成兩條路——今天用的是「查/改我的 HR 資料」這條路（請假、薪資），下一堂 Session 3 才會用到「查公司資料」那條路（廠商、採購單）。兩條路名字都是業務動詞、工具名稱也都差不多長相，容易混在一起，但背後查的資料完全不同、權限概念也不同（今天只能查自己的，Session 3 是有授權的同事都能查到一樣的結果）。

MCP 是什麼（用一個比喻）

MCP（Model Context Protocol）是讓 Codex 可以「打電話給其他系統」的機制。

你問 Codex：「我今年請了幾天年假？」

Codex 不是從記憶回答，而是：

  1. 理解你的問題
  2. 決定要打電話給 list_absences 工具
  3. 把答案帶回來，用你聽得懂的方式說給你聽

現場 Demo（講師操作）：

示範問法一：

我今年請了哪些假？

示範問法二：

我想申請下週一年假，先幫我預覽，不要真的送出。

示範問法三（重要）：

好，我確認了，幫我送出。

觀察重點：

  • 第一個問題：Codex 去查了什麼？回傳什麼？
  • 第二個問題：系統回傳「預覽」而不是直接執行——為什麼？
  • 第三個問題：確認後才真的送出——這個設計代表什麼？

為什麼寫入要有「預覽 → 確認」兩步？

讀資料是安全的，寫資料是有風險的。

如果 Codex 誤解你的意思，直接送出了一個不對的請假申請：

  • 打錯日期
  • 選錯假別
  • 天數錯誤

沒有確認步驟，你不會有機會發現。

「預覽 → 確認」是系統幫你做的最後一道關卡。確認之前，認真看一遍。

這是四堂課的技巧 2／4：安全鍵。 Session 1 的 RCFC 讓你把任務講清楚，但講清楚不代表 Codex 一定沒理解錯——「預覽 → 確認」就是萬一它理解錯了，你還有機會擋下來的最後關卡。之後不管是今天的請假、Session 4 的報價單，看到「這是預覽，要不要送出」，養成習慣：慢下來，認真看一遍再回答。

13:50–14:15 MCP 設定 + 連線驗收（25 分鐘）

學員不用另開人資網站登入。 每人用設定卡上的專屬 Bearer，對應一位示範員工（WDL-Exxx）。身分已綁在 Bearer；查到的假單／薪資是該座位示範員工的資料，不是學員本人。

A. MCP 設定（約 15–18 分鐘）— 先做這段

  1. 發設定卡（一人一張，勿互換 Bearer）
  2. 投影逐步操作；請學員跟著做，先不要開新任務查系統
  3. ChatGPT Work：左側 Plugins → 齒輪 Manage → MCPs → Add server → Streamable HTTP
  4. 貼設定卡上的 URL（結尾要有 /mcp）與 Authorization: Bearer …（含前面的 Bearer ）→ Save → 開關為開；有 Restart 就按
  5. 只開課堂這一台；關掉本機 STDIO／其他業務 MCP
  6. 常見錯：少打 Bearer 、URL 少 /mcp、選成 STDIO、貼到別人的卡、舊對話沒重開

名稱是 wendell-dataverse-cloud（或設定卡上的名字）。

B. 連線驗收（約 5–7 分鐘）— 設定完成後再做

確認 MCP 開關已開，再開一個全新任務（New task），輸入：

請用 platform_health 查系統健康狀態。

預期回應含「所有服務正常」與 HR 員工代碼：WDL-Exxx（應與設定卡一致）。若出現 Pillow／磁碟／本機 Python，或說「沒有 platform_health 工具」，代表還沒連上課堂 MCP——回到 A 段檢查開關／Bearer／Restart，不要急著做練習一。

通用 Plugins（Notion／Slack／信箱／日曆）：有興趣者可在市集自行安裝；本堂主線仍是 ERP MCP，不強制。

15:45–16:00 討論 + 預告（15 分鐘）

討論：AI 寫入 ERP，我們應該信任它嗎？

讓學員說出自己的想法，不要急著給答案。

幾個引導問題：

  1. 「你剛才送出請假的時候，有沒有再看一次預覽？你看了幾秒？」
  2. 「如果系統沒有預覽步驟，你還會這麼放心讓 Codex 送嗎？」
  3. 「哪些 ERP 操作你覺得 Codex 可以直接做？哪些你覺得一定要人確認？」

這個討論沒有標準答案。 目的是讓學員開始思考「AI 做決定」和「AI 協助做決定」的界線。

預告 Session 3

今天我們查的是「我自己的資料」。

下一堂課，我們要查「公司的業務資料」——廠商、採購單、應付帳款。 採購、財務會有最多收穫。 最後也會延伸看一下客戶和委託案。 實驗室同仁也會發現：平日常要問的委託案進度、儀器採購、廠商與庫存，今天這堂還查不到——那些是「公司資料」，下一堂才接上。

課後作業（必做，Session 3 開場分享）

寫下你自己部門每週都要查一次的「公司資料」（不是個人的假單薪資，是業務/採購/財務/實驗室進度這類資料），用 RCFC 想好你會怎麼問 Codex，先寫下問法草稿（還不用真的查得到，Session 3 會用示範環境驗證）。

帶來下堂課的東西：

  1. 我每週要查的資料是什麼
  2. 我會怎麼問（用 RCFC 寫一段任務描述）
  3. 如果查得到，我省下的時間大概是多少

本堂講義重點（帶回去用）

Codex 單純 vs. Codex + 工具

單純 Codex Codex + MCP 工具
資料來源 訓練資料 即時連線外部系統
適合做什麼 語言、翻譯、寫作、分析文件 查詢 ERP 資料、提交申請

安全鍵：預覽 → 確認

  1. 送出查詢／申請前，Codex 先給「預覽」，不會直接執行
  2. 認真看一遍預覽內容（日期、天數、假別、金額有沒有錯）
  3. 確認無誤才回答「確認，送出」
  4. 讀資料安全，寫資料一定要走這兩步——沒有例外

今天用過的示範問法（回去可以照樣造句）：

我今年請了哪些假？
我有哪些薪資單可以查？
我上個月的薪資明細是什麼？
幫我查 2026 年 6 月的薪資明細。
幫我查薪資單 PAY-2026-06 的內容。
我想申請下週一年假，先幫我預覽，不要真的送出。
